## M1: callLimit 和 quota 字段未检查

### 问题描述

`apiKey.callLimit` 和 `apiKey.quota` 字段在 Admin API 中可配置，在 Schema 中存在，但在请求处理管道中**从未被检查**。管理员设置了限制但实际不生效。

### 修复必要性

**高** — 这是功能承诺但未实现的问题。管理员设置 callLimit 后误以为有效，可能导致资源超用。

### 修复方案

#### M1a: callLimit 检查

**需要新增**: ApiKey 模型增加 `callCount` 字段（已调用次数），或在请求处理时用 `requestLog` 统计。

**推荐方案**: 使用 `requestLog` 表统计（无需 Schema 变更）

```typescript
// 在 API Key 验证后、路由选择前插入
const effectiveCallLimit = apiKey.callLimit ?? apiKey.plan?.callLimit ?? null;
if (effectiveCallLimit !== null) {
  const callCount = await prisma.requestLog.count({
    where: { keyId: apiKey.id },
  });
  if (callCount >= effectiveCallLimit) {
    return Response.json(
      { error: { message: "API Key 调用次数已达上限", type: "invalid_request_error" } },
      { status: 403 }
    );
  }
}
```

**性能考量**: `requestLog.count()` 在大数据量下可能较慢。优化方案：
- 短期：使用 `requestLog.count()`（有 `keyId` 索引，百万级数据 < 50ms）
- 长期：ApiKey 模型增加 `callCount` 字段，请求完成时原子递增（与 `usedTokens` 模式一致）

#### M1b: quota 检查

**已有 `tokenLimit` 检查**，`quota` 字段语义与 `tokenQuota` 重叠。需确认 `quota` 的用途：

- 如果 `quota` = 自定义 token 额度 → 已被 `tokenLimit` 覆盖（`effectiveTokenLimit = apiKey.tokenLimit ?? apiKey.plan?.tokenQuota`）
- 如果 `quota` = 自定义金额/积分额度 → 需要额外的计费逻辑

**推荐方案**: 如果 `quota` 与 `tokenQuota` 语义相同，则无需额外检查（已被 tokenLimit 覆盖）。如果不同，需在 Schema 和业务逻辑中明确定义。

### 影响范围

- `src/app/api/v1/chat/completions/route.ts`
- `src/app/api/v1/completions/route.ts`
- 可能需要 Schema 变更（增加 `callCount` 字段）

### 潜在风险

- **性能**: 每次请求增加一次 DB 查询（count），在高并发下可能成为瓶颈
- **缓解**: 使用内存计数器 + 定期同步 DB，或增加 `callCount` 字段原子递增

## M2: resetPeriod 无自动重置机制

### 问题描述

`apiKey.resetPeriod` 字段支持 "monthly" / "daily" / "never"，但没有任何机制根据该字段自动重置 `usedTokens`。

### 修复必要性

**中高** — 设了 monthly reset 的 Key，`usedTokens` 永远不会归零，导致 tokenLimit 永久锁定。

### 修复方案

#### 方案 A: 后台定时任务（推荐）

在 `src/instrumentation.ts` 的 `register()` 中启动定时任务，每小时检查一次：

```typescript
// 在 register() 中与 initializeAdmin()、startProxyHealthChecker() 并列
try {
  const { startApiKeyResetScheduler } = await import("./lib/api-key-reset");
  startApiKeyResetScheduler();
} catch (error) {
  console.error("[instrumentation] API Key 重置调度器启动失败:", error);
}
```

新建 `src/lib/api-key-reset.ts`:

```typescript
import { prisma } from "./prisma";

let timer: ReturnType<typeof setInterval> | null = null;
const CHECK_INTERVAL_MS = 60 * 60 * 1000; // 每小时检查一次

function shouldReset(apiKey: { resetPeriod: string | null; updatedAt: Date }): boolean {
  const now = new Date();
  const updated = new Date(apiKey.updatedAt);

  switch (apiKey.resetPeriod) {
    case "daily":
      // 上次更新在昨天或更早
      return updated.toDateString() !== now.toDateString();
    case "monthly":
      // 上次更新在上个月或更早
      return updated.getMonth() !== now.getMonth() || updated.getFullYear() !== now.getFullYear();
    case "never":
    default:
      return false;
  }
}

async function resetUsage() {
  try {
    const keysToReset = await prisma.apiKey.findMany({
      where: {
        resetPeriod: { not: "never" },
        status: "active",
      },
    });

    let resetCount = 0;
    for (const key of keysToReset) {
      if (shouldReset(key)) {
        await prisma.apiKey.update({
          where: { id: key.id },
          data: { usedTokens: 0 },
        });
        resetCount++;
      }
    }

    if (resetCount > 0) {
      console.log(`[api-key-reset] 已重置 ${resetCount} 个 API Key 的用量`);
    }
  } catch (err) {
    console.error("[api-key-reset] 重置异常:", err instanceof Error ? err.message : String(err));
  }
}

export function startApiKeyResetScheduler() {
  if (timer) return;
  console.log("[api-key-reset] 启动 API Key 用量重置调度器，间隔 1 小时");
  setTimeout(resetUsage, 60_000); // 首次延迟 1 分钟
  timer = setInterval(resetUsage, CHECK_INTERVAL_MS);
}
```

#### 方案 B: 请求时惰性重置

在每次请求时检查是否需要重置（避免后台任务）：

```typescript
// 在 API Key 验证后
if (apiKey.resetPeriod && apiKey.resetPeriod !== "never") {
  const now = new Date();
  const updated = new Date(apiKey.updatedAt);
  let needsReset = false;

  if (apiKey.resetPeriod === "daily" && updated.toDateString() !== now.toDateString()) {
    needsReset = true;
  } else if (apiKey.resetPeriod === "monthly" && 
    (updated.getMonth() !== now.getMonth() || updated.getFullYear() !== now.getFullYear())) {
    needsReset = true;
  }

  if (needsReset) {
    await prisma.apiKey.update({
      where: { id: apiKey.id },
      data: { usedTokens: 0 },
    });
    apiKey.usedTokens = BigInt(0); // 更新内存中的值
  }
}
```

**推荐方案 A**（后台定时任务），理由：
1. 惰性重置导致第一个请求的延迟增加
2. 后台任务可以批量处理，减少 DB 连接数
3. 与 `startProxyHealthChecker()` 模式一致

### 影响范围

- 新建 `src/lib/api-key-reset.ts`
- 修改 `src/instrumentation.ts`（添加调度器启动）

### 潜在风险

- **低风险**: 定时任务仅在非生产环境的测试中可能因 `Date` 不一致导致误重置
- **缓解**: 使用 `updatedAt` 字段判断而非绝对时间

## M3: API Key 级 RPM/TPM 限制未执行

### 问题描述

`apiKey.rpmLimit` 和 `apiKey.tpmLimit` 字段在 Schema 中存在，但请求处理中**仅检查平台级限制**（`platform.rpmLimit` / `platform.tpmLimit`），Key 级限制从未执行。

### 修复必要性

**中高** — 管理员为单个 Key 设置了 RPM/TPM 限制但不生效，可能导致单个用户占用过多平台配额。

### 修复方案

在现有 `checkPlatformRateLimit()` 基础上，增加 Key 级别的速率限制。需要为每个 API Key 维护独立的窗口计数器。

#### 新增 Key 级速率限制函数

在 `src/lib/rate-limiter.ts` 中新增：

```typescript
const keyWindows = new Map<string, WindowEntry>();

/**
 * 检查 API Key 速率限制（RPM + TPM）
 */
export function checkKeyRateLimit(
  keyId: string,
  rpmLimit: number | null,
  tpmLimit: number | null,
  tokenCount: number = 0
): RateLimitResult {
  if (tokenCount < 0) tokenCount = 0;
  const now = Date.now();
  const windowMs = 60_000;

  let entry = keyWindows.get(keyId);

  if (!entry || now - entry.windowStart >= windowMs) {
    entry = { count: 0, tokens: 0, windowStart: now };
    keyWindows.set(keyId, entry);
  }

  if (rpmLimit !== null && entry.count >= rpmLimit) {
    return { allowed: false, remaining: 0, resetAt: entry.windowStart + windowMs };
  }

  if (tpmLimit !== null && entry.tokens + tokenCount >= tpmLimit) {
    return { allowed: false, remaining: 0, resetAt: entry.windowStart + windowMs };
  }

  entry.count += 1;
  entry.tokens += tokenCount;

  return {
    allowed: true,
    remaining: rpmLimit !== null ? rpmLimit - entry.count : Infinity,
    resetAt: entry.windowStart + windowMs,
  };
}
```

#### 在请求处理中调用

在两个 route 文件的 API Key 验证后、路由选择前插入：

```typescript
// 5. API Key 级速率限制
const effectiveRpmLimit = apiKey.rpmLimit ?? apiKey.plan?.rpmLimit ?? null;
const effectiveTpmLimit = apiKey.tpmLimit ?? apiKey.plan?.tpmLimit ?? null;

const keyRateResult = checkKeyRateLimit(
  apiKey.id,
  effectiveRpmLimit,
  effectiveTpmLimit,
  0 // 预检阶段 tokenCount=0，实际 token 在请求完成后更新
);

if (!keyRateResult.allowed) {
  return Response.json(
    {
      error: {
        message: "API Key 请求速率超过限制，请稍后重试",
        type: "rate_limit_error",
      },
    },
    {
      status: 429,
      headers: {
        "X-RateLimit-Reset": String(Math.ceil(keyRateResult.resetAt / 1000)),
      },
    }
  );
}
```

### 影响范围

- `src/lib/rate-limiter.ts`（新增 `checkKeyRateLimit` 函数）
- `src/app/api/v1/chat/completions/route.ts`
- `src/app/api/v1/completions/route.ts`

### 潜在风险

- **内存占用**: 每个活跃 API Key 占用一个 Map 条目（~100 bytes），万级 Key 仅 ~1MB，可接受
- **多实例部署**: 内存计数器在多实例间不共享，需要 Redis（当前项目单实例，暂不需要）
- **清理机制**: 需要在现有 `startRateLimitCleanup()` 中同时清理 Key 窗口

### 实施顺序建议

1. **M1 + M2**（P1，可并行实施）

   - M2 需要新建文件 + instrumentation 修改
2. **M3**（P2，可独立实施）
   - 改动较大但风险可控
   - 建议在 M1/M2 稳定后再实施

### 测试策略

- 单元测试：为 `checkKeyRateLimit()` 和 `shouldReset()` 编写测试
